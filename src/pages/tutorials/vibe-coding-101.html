MODULE_03

FRAMEWORK A.D.A.P.T.

PASO · A
Architect First
Antes de abrir Cursor o Claude, dibuja el sistema. ¿Qué entidades existen? ¿Qué flujos hay? Un diagrama de 10 min te ahorra 3h de refactoring.
"Tengo Users, Projects, y Assets. Users crean Projects. Projects tienen múltiples Assets con estados: draft → processing → published."
PASO · D
Define el PRD
Product Requirements Document. No un essay — un documento estructurado con: contexto de negocio, user stories, stack declarado, y criterios de éxito medibles.
"Como usuario autenticado, quiero subir un asset de imagen, para que el sistema genere 3 versiones optimizadas. Criterio: procesamiento < 30s."
PASO · A
Assign to AI
Con el PRD listo, das instrucciones en chunks manejables. No "buildea mi app" — "construye el componente ImageUploader con estos requerimientos específicos".
"Construye POST /assets/upload. Stack: Next.js 14, TypeScript, Supabase Storage. Valida tipo de archivo, sube, retorna URL público."
PASO · P
Probe & Test
Prueba el código generado. No confíes ciegamente. Revisa edge cases, manejo de errores, seguridad. Pregunta: "¿Qué podría fallar aquí?"
"No hay validación del tamaño del archivo. Agrega un límite de 10MB y retorna error 413 con mensaje claro si se excede."
PASO · T
Tune & Iterate
Refina conversacionalmente. La primera versión raramente es la final. Itera con contexto acumulado. Documenta en tu PRD — vive y evoluciona.
"Integra este endpoint con el componente ImageUploader que creamos. Mantén el mismo patrón de manejo de errores y estado de loading."

MODULE_05

TRAINING PROTOCOLS

01
Escribir PRDs que la AI pueda ejecutar
~15 MIN

Un PRD para AI no es un documento bonito — es un contrato de instrucciones. Cuanto más específico, mejor el output.

▷ 01Sección de Contexto: ¿Qué problema resuelve? ¿Para quién? ¿Por qué ahora?
▷ 02Stack declarado: Next.js 14, TypeScript, Tailwind, Supabase — siempre explícito
▷ 03User Stories: "Como [rol], quiero [acción], para [beneficio]"
▷ 04Criterios medibles: No "funciona bien" — "carga en menos de 2 segundos"
// PRD TEMPLATE — feature section

FEATURE: Dashboard de Proyectos
STACK: Next.js 14 · TypeScript · Tailwind · Supabase
USER_STORY: "Como PM, quiero ver proyectos activos con estado y actividad."
INPUTS: userId autenticado via Supabase Auth
OUTPUT: Grid de cards: nombre, status, client, updated_at
CONSTRAINTS: 3 estados: active | paused | done · sort: updated_at DESC
SUCCESS: Carga <1.5s · Empty state si no hay proyectos
02
Decomposición: De visión a tareas atómicas
~20 MIN

El error más común: pedirle a la AI "construye la app". La AI necesita tareas pequeñas, concretas, con contexto claro.

▷ 01Epic → Feature → Task: Descompón en 3 niveles antes de promptear
▷ 02Una tarea = un componente: "Construye el hook useProjects" no "construye el dashboard"
▷ 03Incluye dependencias: "Este hook usa el Supabase client en lib/supabase.ts"
▷ 04Define el contrato: inputs, outputs, y errores esperados antes de pedir código
03
Mantener coherencia a través de sesiones
~15 MIN

La AI no tiene memoria entre sesiones. Tú eres la memoria del proyecto. El claude.md y el PRD son el sistema nervioso que mantiene la coherencia.

▷ 01claude.md obligatorio: Stack, convenciones de naming, estructura de carpetas, patrones usados
▷ 02Session starter: "Lee el claude.md y el PRD antes de continuar. Seguimos desde: [contexto]"
▷ 03Decisiones documentadas: ¿Por qué Supabase y no Firebase? Escríbelo. La AI lo necesita.
04
Debugging como Arquitecto — no línea por línea
~25 MIN

El arquitecto debuggea el sistema, no líneas. ¿En qué capa está el problema? ¿Datos, lógica, UI, network? Localiza la capa antes de ejecutar cualquier fix.

▷ 01Describe el síntoma: "El usuario ve datos vacíos aunque hay datos en DB. El fetch retorna array vacío."
▷ 02Pide root cause: "Antes de fixear, explícame por qué crees que está pasando esto"
▷ 03Valida side effects: "¿Este fix afecta otros componentes? ¿Qué más podría romperse?"
▷ 04Documenta el bug: Agrega al claude.md como decisión tomada. Evita reincidir.
05
Comunicar valor al cliente sin hablar de código
~10 MIN

El arquitecto habla en outcomes, no en outputs. Tu cliente no paga por componentes React — paga por leads, eficiencia, y crecimiento.

✗ MAL"Implementé un hook con React Query para data fetching optimizado con caché"
✓ BIEN"El dashboard carga 3× más rápido — tus clientes llegan a la acción clave en <2 segundos"
▷ PRDEl PRD es tu contrato: muestra qué se definió y qué se entregó. Transparencia = confianza = retención.

PRIMER MISSION

OTTO // Feature "Asset Upload" — PRD Exercise
Usando OTTO como proyecto real de Denysoft, escribe el PRD de un feature específico y promptea a Claude para que lo construya. Este ejercicio une los 4 sombreros en un flujo real.
01
Elige UN feature pequeño de OTTO — por ejemplo: "Guardar prompts de AI en un proyecto específico". Pequeño y concreto.
02
Escribe el PRD en Notion usando el template aprendido: contexto, stack, user story, inputs/outputs, criterios de éxito. Máximo 1 página.
03
Decompon en 3 tareas atómicas: 1) tabla en Supabase, 2) API endpoint, 3) componente UI.
04
Promptea Claude o Cursor con el PRD como contexto. Empieza por la tarea 1. Evalúa como QA crítico antes de pasar a la 2.
05
Documenta el resultado: ¿Qué funcionó? ¿Qué corregiste? ¿Qué agregas al claude.md de OTTO? Eso es pensar como arquitecto.
