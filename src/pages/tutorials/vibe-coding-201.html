ARCHITECT FIRST
Antes de Abrir el Editor

1A
ARCHITECT THINKING
Entender el Negocio Antes de Pensar en Código
¿Qué necesita realmente el cliente?
DÍA 1

El Arquitecto hace preguntas de negocio antes de decidir cualquier cosa técnica. No preguntas de diseño — preguntas de outcomes. ¿Qué define el éxito? ¿Quién es el usuario real? ¿Cuál es la acción más importante de todo el site?

PREGUNTA 01: ¿Cuál es el único objetivo de este website?
→ Generar leads calificados de empresas B2B que buscan automatización

PREGUNTA 02: ¿Quién visita este site y qué busca?
→ Gerentes de operaciones 35-55 años, frustrados con procesos manuales, buscando solución confiable

PREGUNTA 03: ¿Cuál es la ONE action que queremos que haga cada visitante?
→ Agendar una llamada de diagnóstico gratuito (formulario de lead)

PREGUNTA 04: ¿Cómo sabremos que funcionó?
→ KPI principal: tasa de conversión visitante → lead agendado > 3%

→ INSIGHT El Mapa Mental del Arquitecto
▷ KEYTodo el site sirve UNA cosa: llevar al visitante a apretar el botón de "Agenda tu diagnóstico". Cada sección, cada texto, cada imagen debe servir eso.
▷ KEYEl pain point es más poderoso que el feature: "¿Cuánto te cuesta hacer eso manualmente hoy?" convierte mejor que "Nuestro sistema tiene IA".
▷ KEYMobile-first no es opcional: El 60%+ del tráfico B2B llega de mobile. Si el formulario no funciona en iPhone, perdiste el lead.

⚠ No empieces a definir páginas ni componentes todavía. Primero entiende el negocio. Un website bonito con la arquitectura de contenido equivocada convierte 0%. Ocurre todo el tiempo.

1C
TECH DECISIONS
Stack Selection — Por Qué Cada Herramienta
Decisiones del arquitecto con justificación
DÍA 2

El Arquitecto no elige tecnología por moda — la elige por fit con los constraints del proyecto. Budget limitado, timeline de 3 semanas, cliente no técnico. Cada decisión tiene un por qué.

CAPATECNOLOGÍAPOR QUÉ — NO OTRA
FRAMEWORK NEXT.JS 14 App Router + SSR nativo = SEO perfecto. La AI (Cursor/Claude) conoce Next.js más que cualquier otro framework. Máximo output.
DATABASE + AUTH SUPABASE PostgreSQL + Auth + Storage + Realtime en uno. El dashboard de admin usa Supabase Auth sin escribir una línea de auth custom. No Firebase: menos vendor lock-in, SQL real, mejor precio.
EMAIL RESEND API simple, deliverability excelente, free tier generoso (100 emails/día). Para confirmaciones de lead y notificaciones al equipo. No SendGrid: overkill para este scope.
STYLING TAILWIND CSS La AI genera Tailwind con precisión. Rapid iteration sin escribir CSS custom. Consistencia por defecto.
DEPLOY VERCEL Deploy de Next.js en segundos. CDN global incluido. Preview deployments por branch. Analytics básico gratis.
FORMULARIOS REACT HOOK FORM Zero re-renders. Validación integrada. La AI lo conoce perfectamente. No Formik: más pesado y verbose.
VALIDACIÓN ZOD TypeScript-first. Mismo schema en frontend y API route = single source of truth. Previene data inconsistente desde el diseño.

// Lo que decides TÚ, no la AI:
Tú decides: Stack, integraciones, modelo de datos, arquitectura
AI decide: Implementación específica, código, patrones internos

// Si la AI te sugiere cambiar el stack, pregúntate:
¿Cambia por una razón técnica válida o porque es lo que entrenó más?
Tú tienes la última palabra siempre.

FASE_03

BUILD WITH AI
Prompts Reales para Tareas Reales

3A
FOUNDATION FIRST
Setup Inicial — Lo que Buildeas Tú, No la AI
El Arquitecto configura la base manualmente
DÍA 4

Hay cosas que tú configuras manualmente porque involucran credenciales, variables de entorno, y decisiones de seguridad que la AI no debe manejar.

▷ 01npx create-next-app@latest nexaflow-web — TypeScript: Yes, Tailwind: Yes, App Router: Yes, src/ directory: No
▷ 02Crear proyecto en Supabase → copiar URL + anon key → crear tabla leads con el schema definido
▷ 03Crear cuenta Resend → verificar dominio del cliente → obtener API key
▷ 04Crear .env.local con todas las variables: NEXT_PUBLIC_SUPABASE_URL, SUPABASE_SERVICE_KEY, RESEND_API_KEY
▷ 05Crear .env.example sin valores reales → commitearlo al repo para documentar qué variables existen
▷ 06Crear claude.md en la raíz del proyecto → pegar el PRD completo + stack + convenciones

⚠ Nunca le des tus API keys reales a la AI en el prompt. El claude.md solo documenta el NOMBRE de las variables (.env.local), no sus valores. La AI las lee del proceso como process.env.VARIABLE_NAME.

3B
AI EXECUTION — PROMPTS REALES
Cómo Pedir Cada Componente a la AI
Prompts exactos con contexto, constraints, y outputs
DÍA 5-11

Aquí está la diferencia entre el vibe coder promedio y el Arquitecto: la calidad del prompt determina la calidad del código. Estos son los prompts reales, en el orden correcto.

PROMPT 01 Supabase Client + Types

Eres un senior developer en Next.js 14 con App Router y TypeScript.

Lee el claude.md adjunto — es el PRD completo del proyecto.

Primera tarea: Crea el Supabase client para este proyecto.
- Archivo: lib/supabase/client.ts → client-side (createBrowserClient)
- Archivo: lib/supabase/server.ts → server-side (createServerClient con cookies)
- Archivo: types/database.ts → TypeScript types para la tabla 'leads' según el schema del PRD

Usa @supabase/ssr (NO @supabase/auth-helpers que está deprecated).
Las env vars son: NEXT_PUBLIC_SUPABASE_URL y NEXT_PUBLIC_SUPABASE_ANON_KEY
Incluye los tipos exactos del schema de leads del PRD.

Deberías recibir:

  • lib/supabase/client.ts con createBrowserClient
  • lib/supabase/server.ts con createServerClient usando cookies()
  • types/database.ts con interface Lead y LeadInsert
PROMPT 02 Zod Schema + Validación del Form

Continuando con el proyecto NexaFlow. El Supabase client ya está creado.

Crea el Zod schema de validación para el LeadForm:
- Archivo: lib/validations/lead.ts
- Campos según el PRD: name (min 2), email (válido), company (min 2), phone (opcional, E.164 format), urgency (enum: immediate/soon/exploring), message (opcional, max 500)
- Exporta: leadSchema (Zod), LeadFormData (TypeScript type inferido del schema)
- Este mismo schema se usará en el frontend (react-hook-form) y en la API route (validación server-side)
Single source of truth — no duplicar lógica de validación.

PROMPT 03 API Route — Submit Lead

Crea la API route para recibir y procesar el form de lead:
- Archivo: app/api/leads/route.ts
- Método: POST
- Flujo: 1) Validar body con leadSchema (Zod) 2) Guardar en Supabase tabla 'leads' con server client 3) Enviar email de confirmación al lead via Resend 4) Enviar email de notificación al equipo (EMAIL_TEAM en .env) 5) Retornar { success: true, id: lead.id }
- Error handling: try/catch, retorna errores estructurados { success: false, error: string }
- Rate limiting básico: no más de 1 lead por email en 24h (query a Supabase antes de insertar)
- Env vars usadas: SUPABASE_SERVICE_KEY (no anon), RESEND_API_KEY, EMAIL_FROM, EMAIL_TEAM
Los templates de email serán simples strings HTML por ahora — los mejoraremos después.

PROMPT 04 LeadForm Component

Crea el componente LeadForm:
- Archivo: components/forms/LeadForm.tsx
- Client component ('use client')
- Usa react-hook-form con zodResolver conectado a leadSchema de lib/validations/lead.ts
- Campos: nombre, email, empresa, teléfono (opcional), urgencia (select), mensaje (textarea opcional)
- Estados: idle → loading → success | error
- En success: muestra mensaje de confirmación inline (NO redirect, el parent maneja la navegación)
- Styling: Tailwind, diseño limpio profesional B2B, botón de submit con loading spinner
- Conecta con POST /api/leads usando fetch, maneja errores con toast o inline message
- Accesibilidad: labels correctos, aria-describedby para errores, focus management

⚠ Después de cada componente: léelo, entiéndelo, pruébalo. No avances al siguiente prompt hasta que el anterior funcione. La deuda técnica se acumula exponencialmente cuando stackeas componentes sobre código que no has verificado.

PROMPT 05 Hero Section — El Más Crítico

Crea el HeroSection para NexaFlow — este es el componente más importante del site:
- Archivo: components/sections/HeroSection.tsx
- Headline principal: "Automatiza lo que te está costando tiempo y dinero hoy" (editable via props)
- Subline: propuesta de valor en 2 líneas máximo
- CTA principal: botón "Agenda tu Diagnóstico Gratuito" → scroll smooth a #lead-form
- CTA secundario: "Ver casos de éxito" → link a /casos
- Trust indicators: 3 números clave (ej: "47 procesos automatizados", "12 clientes", "3.2h ahorradas/día en promedio")
- Background: gradiente sutil o imagen hero con overlay, NO blanco plano
- Diseño B2B profesional: NO startup genérica, NO purple gradients. Confiable, serio, results-focused
- Full viewport height en desktop, auto en mobile
- Incluye structured data JSON-LD para Organization schema

3C
ADMIN DASHBOARD
Dashboard de Leads — La Herramienta del Cliente
El cliente necesita ver sus leads desde día 1
DÍA 10-11

El admin dashboard es lo que hace que el cliente perciba el valor inmediatamente. No es opcional — es la prueba de que el sistema funciona. Buildéalo en los últimos días del sprint.

Crea el Admin Dashboard para ver y gestionar leads:
- Ruta: app/admin/page.tsx (Server Component)
- Protección: middleware.ts verifica sesión Supabase Auth, redirect a /admin/login si no hay sesión
- Ruta login: app/admin/login/page.tsx → magic link auth con Supabase (no password)
- Dashboard muestra: tabla de leads con columnas: nombre, email, empresa, urgencia, status, fecha
- Filtros: por status (new/contacted/qualified/closed), por urgencia, búsqueda por nombre/empresa
- Cada fila: click abre un panel lateral (drawer) con todos los datos del lead
- Desde el panel: puede cambiar el status del lead (Server Action)
- Stats header: total leads, leads nuevos hoy, leads esta semana, tasa de conversión estimada
- Diseño: limpio, funcional, no necesita ser bonito — necesita ser rápido y claro
- Export CSV: botón que descarga todos los leads como CSV (Client Component)

// Por qué magic link y no password:
RAZÓN 1: No gestionar passwords del cliente = menos soporte
RAZÓN 2: Más seguro — no hay password que hackear
RAZÓN 3: Supabase lo maneja completo, 0 código de auth custom
RAZÓN 4: El cliente solo necesita su email — experiencia simple

NEGOCIO: El email de confirmación llega al lead en menos de 30 segundos

NEGOCIO: El lead aparece en el admin dashboard con todos sus datos correctos

LAUNCH & BEYOND
Deploy, Entrega, y Siguiente Nivel

5
DEPLOY PROTOCOL
Launch en Vercel — El Proceso Correcto
Arquitecto controla el deploy, no lo improvisa
DÍA 18-19

Deploy no es "subir el código y rezar". El Arquitecto tiene un protocolo de launch que asegura que nada está roto en producción antes de dar el dominio al cliente.

▷ 01Preview Deploy primero: Vercel genera un URL de preview por cada push. Prueba todo en el preview URL antes de asignar el dominio.
▷ 02Variables de entorno en Vercel: Settings → Environment Variables → agregar todas las del .env. Verificar que SUPABASE_SERVICE_KEY solo esté en Server (no en Edge ni Client).
▷ 03Dominio del cliente: Vercel → Domains → Add domain. DNS CNAME/A record según instrucciones. SSL automático. Espera propagación 15-60 min.
▷ 04Prueba en producción con lead real: Tú envías un lead de prueba en el dominio final. Verificas: guardado en Supabase, emails enviados, aparece en admin.
▷ 05Google Search Console: Registrar el dominio, subir sitemap.xml. Esto inicia el indexing SEO desde el día 1.
▷ 06Entrega al cliente: Un PDF de 1 página con: URL del site, URL del admin (/admin), email de login (magic link), cómo cambiar el status de un lead. Simple.
